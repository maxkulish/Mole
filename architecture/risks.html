Risk Cards (sorted by severity)
Sudo for system cache cleanup MEDIUM

Description: Shell scripts request sudo to delete system caches and logs in /private/var/ and related paths. If the path validation in lib/core/file_ops.sh had a bypass bug, a privileged deletion could remove important system files.

Evidence: lib/core/sudo.sh, lib/core/file_ops.sh:67-176 (validate_path_for_deletion), lib/core/file_ops.sh:303 (safe_sudo_remove)

Mitigation: Path validation has multiple guards (empty path, relative path, traversal, control chars, system dir blocklist). An allowlist of safe sudo paths is maintained. sudo sessions are time-limited (5 min). Dry-run mode available for preview.

Installer downloads from GitHub (install.sh) MEDIUM

Description: The installer downloads binaries from GitHub releases. If the GitHub repository or release were compromised (supply-chain attack), a user running the installer could get a malicious binary. This is standard for many open-source tools but worth noting.

Evidence: install.sh:496 downloads binary from https://github.com/tw93/mole/releases/download/V${version}/

Mitigation: Downloads are over HTTPS from the official GitHub repository. Version validation is performed before download. For higher assurance, build from source using go build ./cmd/....

Shell eval usage in base.sh LOW

Description: Limited use of eval in shell scripts for home directory expansion and trap cleanup registration. If the variables passed to eval contained user-controlled content, this could be an injection vector.

Evidence: lib/core/base.sh:256, 690, lib/clean/project.sh:860-861

Mitigation: Variables passed to eval are derived from internal logic (home directory expansion, trap string construction), not from user input. No user-supplied strings flow into these eval calls.

AppleScript argument escaping LOW

Description: File paths are embedded in an AppleScript string for Finder Trash operations. If escaping were incomplete, a specially crafted filename could inject AppleScript commands.

Evidence: cmd/analyze/delete.go:127-131 - escapes quotes and backslashes

Mitigation: The code explicitly escapes single quotes and backslashes before embedding paths in AppleScript. User-selected paths come from directory scanning (not arbitrary user input). Risk is very low in practice.

Information loss risk (disk analyzer) LOW

Description: The disk analyzer skips certain directories (e.g., .git, node_modules, vendor) and file types for performance. Users may not realize these are excluded from the view.

Evidence: cmd/analyze/constants.go - skip directory and extension lists

Mitigation: Skipped directories are well-known development artifacts unlikely to surprise users. The tool focuses on disk usage, not a complete file inventory.

ebitengine/purego — dynamic library loading LOW

Description: The transitive dependency ebitengine/purego (via gopsutil) uses pure Go dynamic library loading to call macOS CoreFoundation APIs without CGO. This is a less-common pattern and warrants awareness.

Evidence: go.mod - github.com/ebitengine/purego v0.9.1 (indirect)

Mitigation: purego is a well-established library by the Ebiten game engine team. It only loads Apple system frameworks (not third-party DLLs). gopsutil is a widely used, well-audited library.

What This Application Cannot Do
  • Send any data to remote servers (no HTTP client in Go code)
  • Exfiltrate files, clipboard contents, or keychain data
  • Install persistent backdoors (no LaunchAgent/daemon installation)
  • Permanently delete files without using macOS Trash
  • Execute arbitrary shell commands from user input (no shell injection vector)
  • Access other users' files (runs as current user, no privilege escalation beyond sudo for system caches)
  • Listen on network ports or accept remote connections
  • Modify system binaries or system files
  • Capture keystrokes beyond the TUI input handling (bubbletea)
  • Access protected data (Health, Photos, Location) without explicit user permission
Recommendations
  • Always run mo analyze --dry-run (or equivalent) before bulk deletion to preview changes
  • Review ~/.config/mole/operations.log periodically to audit what was cleaned
  • If security-sensitive, build from source rather than using the installer binary
  • Consider adding MO_NO_OPLOG=1 only if log privacy is a concern; the log is valuable for auditing
  • Monitor the GitHub repository for security disclosures given the tool's broad filesystem access
